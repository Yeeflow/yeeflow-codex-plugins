# Application Settings Runtime Baseline

## Scope

This focused runtime baseline proves generated Yeeflow application shell settings for custom navigation, navigation groups, layout modes, header settings, and empty app user groups.

It does not test AI, workflow execution, email delivery, external connections, approval forms, document libraries, permissions, or user group member assignment.

## Generated Packages

The generated packages are local runtime artifacts and are intentionally ignored/uncommitted:

| Package | Layout value | Runtime purpose |
| --- | --- | --- |
| `application-settings-navigation-default.v1.yap` | `default` | Horizontal/default menu under the header |
| `application-settings-navigation-left.v1.yap` | `left` | Vertical menu on the left panel |
| `application-settings-navigation-onheader.v1.yap` | `onheader` | Menu inside the header line |
| `application-settings-navigation-none.v1.yap` | `none` | No custom navigation menu |

All packages were generated by `generate-application-settings-runtime-baseline.mjs` with fresh generated IDs below `System.Int64` max.

## Baseline Model

Each package contains:

| Resource | Purpose |
| --- | --- |
| `Service Requests` data list | Default/opening resource and top-level custom navigation label test |
| `Knowledge Articles` data list | Group child resource navigation test |
| `Operations Overview` dashboard | Top-level resource-name fallback and dashboard open test |

Document libraries and approval forms were intentionally omitted so the runtime test stays focused on application settings.

The navigation model uses:

- root `Data.Item.ListModel.LayoutView` JSON
- `sort[]` top-level menu items
- `Type: "classes"` for a custom group
- `list[]` for group child resources
- `DisplayName: "Requests"` for custom display text
- omitted `DisplayName` for the dashboard resource-name fallback
- `Icon: ""` for no-icon behavior
- no nested groups

The header model uses:

- `attrs.appearance.height = 46`
- `attrs.appearance.hideTitle = true`

The user group model uses:

- `Data.AppGroups[]`
- `{ ID, Name, Description }`
- groups `Support Team` and `Operations Managers`
- group IDs included in `ReplaceIds`
- no members, no user IDs, no emails

## User Groups

The default-layout imported app was inspected in application settings. The `User groups` page opened and showed the two placeholder groups:

- `Operations Managers`
- `Support Team`

Descriptions rendered as placeholder no-member text. No member assignment was created or tested.

## Proof Boundary

Runtime-proven:

- generated packages import and open for layout values `default`, `left`, `onheader`, and `none`
- custom navigation menu renders from `LayoutView.sort[]`
- `Type: "classes"` groups render
- group child resources in `list[]` render and can navigate
- custom `DisplayName` renders as menu text
- omitted `DisplayName` falls back to the resource name
- `Icon: ""` works as a no-icon setting in generated navigation
- `attrs.appearance.height = 46` renders without breaking the app shell
- `attrs.appearance.hideTitle = true` hides the application title
- empty `Data.AppGroups[]` records import and appear in the User groups settings UI

Validation-only:

- app group ID coverage in `ReplaceIds`
- hidden duplicate top-level resource item used only for materialization reachability

Unproven:

- user group member assignment schema
- permission behavior using app groups
- additional header height variants beyond `46`
- document library or approval-form navigation inside this focused baseline
- runtime behavior of process/link menu item types

## Follow-Up

Do not generate user group members until a member-bearing export proves a safe schema. Additional header height variants should remain product-known but not runtime-proven unless tested in a future focused baseline.
