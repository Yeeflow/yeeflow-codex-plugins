Yeeflow Event Product Slide Deck Standard

Purpose
Use this standard for Yeeflow event-facing product introduction slide decks, including decks shown at exhibitions, conferences, partner events, and live customer-facing event presentations.

This standard keeps the deck aligned with:
- Yeeflow’s 2026 product strategy
- Yeeflow’s event booth artwork system
- Yeeflow’s broader design and branding standards

The deck should feel like part of the same family as:
- booth walls
- table-front artwork
- event one-pagers
- sales/event visuals
- product-introduction materials

Core positioning
The deck should present Yeeflow as an enterprise AI application and execution platform.

It should consistently reinforce these strategic themes:
- AI-powered build
- AI-powered use
- AI-powered execution
- platform extensibility
- AI governance and control
- AI work orchestration and reuse

1. Role of the event product deck

The event product deck is not a full product manual.
It is also not a dense implementation proposal.

Its purpose is to:
- introduce Yeeflow clearly and credibly
- explain the platform story quickly
- help event visitors understand category and value
- support event demos and team conversations
- reinforce enterprise trust and product maturity
- create a consistent visual and messaging system across event materials

2. Messaging principles

The deck should sound:
- practical
- enterprise-ready
- governed
- operational
- extensible
- product-led
- credible
- clear

Avoid:
- generic no-code plus AI messaging
- abstract AI transformation language
- hype-heavy future AI messaging
- broad autonomous app-generation claims
- feature dumping without a narrative
- model-count-led storytelling

Preferred messaging themes:
- build business applications faster with AI
- use AI inside real business applications
- execute governed actions across data, workflows, services, and APIs
- connect AI to real business systems
- extend AI through services, integrations, and reusable assets
- govern AI with control, visibility, and discipline

3. Narrative structure

Recommended default event product deck flow:
1. Cover slide
2. Category / positioning slide
3. Why now / business shift slide
4. Platform overview slide
5. AI-powered build slide
6. AI-powered use slide
7. AI-powered execution slide
8. Connected systems / extensibility slide
9. Governance / enterprise readiness slide
10. Current product proof slide
11. Use case / event demo storyline slide
12. Closing / CTA slide

Not every deck must use all slides, but the narrative should remain coherent:
- what Yeeflow is
- why it matters
- how it works
- why it is enterprise-ready
- what is real now
- what action the audience should take next

4. Required slide types

Standardize around these repeatable slide types:
- Cover slide
- Section divider slide
- Hero / positioning slide
- Content slide
- Diagram / architecture slide
- Proof-point slide
- Use-case slide
- Closing slide

Do not create highly inconsistent slide styles from one deck to another.

5. Layout rules

5.1 Content slides
For normal content slides:
- place the Yeeflow logo at bottom-right
- place page number at bottom-left
- reserve the top area for title and optional framing line
- use the main body for one central message
- keep margins generous
- avoid crowding the footer area

5.2 Section divider and opening slides
Section divider slides or opening slides may place the logo at top-left if appropriate.
These slides may be more visual and more brand-forward than standard content slides.

5.3 Body structure
Each content slide should have:
- first read: headline
- second read: short support line or framing statement
- third read: key visual, proof points, or structured explanation

Do not give all elements equal weight.

6. Visual style system

The deck should visually align with Yeeflow’s event and brand materials.

Use:
- Yeeflow blue as the anchor color
- restrained purple and pink accents
- dark premium backgrounds for cover, hero, and section divider slides
- lighter or balanced backgrounds for explanatory slides
- clean spacing
- modern typography hierarchy
- minimal decorative effects
- subtle gradients where needed
- strong contrast and clean readability

Avoid:
- overly noisy tech backgrounds
- excessive glow effects
- dense screenshots on every slide
- decorative AI art
- complex infographic clutter
- inconsistent color systems

7. Background usage rules

7.1 Hero / cover / section slides
These may use:
- dark premium backgrounds
- restrained blue-led gradients
- subtle Yeeflow accent color fields
- more visual drama than content slides, but still controlled

7.2 Explanatory / content slides
These should usually use:
- lighter or balanced backgrounds
- cleaner reading surfaces
- reduced visual intensity
- more whitespace

The full deck should not feel uniformly heavy or overly dark.
Create visual rhythm by balancing dark brand moments with cleaner explanatory slides.

8. Typography and content density

Use large readable titles.
Support lines should be short and meaningful.

Content-density rules:
- one main idea per slide
- one clear headline
- one short support line if needed
- 3 to 5 proof points preferred
- simplify diagrams for scan reading
- limit long paragraphs
- reduce redundant labels

The deck should be easy to scan quickly in an event setting.

9. Diagram and architecture slide rules

Architecture and system slides should:
- show platform logic clearly
- avoid turning into crowded technical schematics
- focus on business-relevant structure
- group related concepts
- emphasize platform connection and execution

Preferred enterprise architecture themes:
- applications
- workflows
- data
- services
- APIs and integrations
- external systems
- governance
- reuse and portability

10. Screenshot usage rules

Use screenshots only when they add proof.
Do not use screenshots as filler.

When screenshots are used:
- keep them large enough to be useful
- simplify surrounding content
- avoid too many screenshots on one slide
- frame them as evidence of product capability, not decoration

11. Event-proof and roadmap realism rules

The deck must stay realistic to Yeeflow’s 2026 maturity path.

Current-event-facing messaging should mainly reflect:
- Q2 product reality
- already shipped proof
- near-term believable platform direction

Good current proof themes include:
- AI Builder practical creation progress
- custom AI connections
- broader model flexibility
- HTTP API and OAuth connection support
- Call HTTP Request for Agent and Copilot
- portability and reuse of AI assets
- AI-assisted coding and extensibility

Do not present:
- unrestricted autonomous enterprise app generation
- late-year roadmap items as if already shipping
- governance or orchestration maturity beyond realistic timeline
- vague future-state claims without product grounding

12. Recommended slide-by-slide message direction

12.1 Cover slide
Purpose:
- establish brand
- establish category
- create premium first impression

Use:
- strong title
- short support line
- large visual anchor
- minimal extra detail

12.2 Category / positioning slide
Purpose:
- explain what Yeeflow is

Typical message direction:
- enterprise AI application and execution platform
- governed AI for real business systems
- build, use, and execute on one platform

12.3 Why now slide
Purpose:
- explain the business change / AI shift

Typical themes:
- AI must move beyond chat
- organizations need operational AI
- connected and governed execution matters

12.4 Platform overview slide
Purpose:
- show the full Yeeflow platform story simply

Typical structure:
- build
- use
- execute
- govern
- extend
- reuse

12.5 Build / Use / Execute slides
Purpose:
- break the product story into digestible sections

These slides should stay practical and not overclaim maturity.

12.6 Connected systems / extensibility slide
Purpose:
- show Yeeflow working across APIs, services, systems, and data

12.7 Governance / enterprise readiness slide
Purpose:
- reinforce trust, control, observability, and enterprise fit

12.8 Current product proof slide
Purpose:
- ground the deck in what exists now

12.9 Use-case / event demo storyline slide
Purpose:
- help event teams tell a believable scenario

12.10 Closing / CTA slide
Purpose:
- reinforce memory
- support next step such as demo, meeting, or follow-up discussion

13. Footer and page treatment

Standard content slides:
- bottom-left: page number
- bottom-right: Yeeflow logo
- keep footer treatment quiet and consistent

Avoid:
- moving these elements slide-by-slide without reason
- oversized footer branding
- multiple competing footer items

14. Logo rules

Use official Yeeflow logo assets only.

Rules:
- dark backgrounds: use the white full logo
- light backgrounds: use the standard full logo
- do not redraw the logo
- do not restyle or decorate the logo
- do not use low-fidelity generated substitutes
- maintain safe spacing around the logo

15. Consistency rule across event materials

The slide deck should visually feel connected to:
- the booth hero wall
- the proof wall
- the operating model wall
- the table front
- event one-pagers and supporting leave-behinds

That means:
- same visual tone
- same color discipline
- same enterprise positioning
- same build / use / execute logic
- same governance and connected-systems narrative

16. Final decision rule

When creating an event product deck:
- prioritize clarity over feature quantity
- prioritize enterprise credibility over hype
- prioritize one coherent platform story over isolated slides
- prioritize realistic product proof over visionary overreach

A strong Yeeflow event deck should feel:
- premium
- calm
- structured
- strategically coherent
- visually consistent with the event system
- credible to enterprise visitors within a few minutes
