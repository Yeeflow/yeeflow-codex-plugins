Service Proposal Standard

Purpose
Use this standard for Yeeflow service-team proposal documents.
Apply the Yeeflow brand, design, and messaging system consistently.

Primary objective
Create proposal documents that are:
- clear
- structured
- enterprise-ready
- customer-relevant
- realistic
- easy to review
- easy to reuse

The proposal should feel:
- modern
- calm
- structured
- trustworthy
- operational
- polished
- business-focused

The proposal should not feel:
- generic
- overdesigned
- vague
- hype-heavy
- overly technical without business framing
- disconnected from Yeeflow’s actual roadmap maturity

Brand and positioning rules
The proposal must stay aligned with Yeeflow’s 2026 positioning:
- enterprise AI application and execution platform
- AI-powered build
- AI-powered use
- AI-powered execution
- platform extensibility
- AI governance and control
- AI work orchestration and reuse

Do not lead with:
- generic no-code messaging
- abstract AI transformation language
- unrestricted autonomy claims
- roadmap promises that are not yet realistic

Tone rules
The writing should be:
- direct
- professional
- concise
- practical
- confident but credible

Use language that connects capability to operational business value.

Default proposal structure
1. Title page or proposal header
- customer name
- proposal title
- date
- prepared by

2. Executive summary
- customer context
- core business need
- summary of recommended Yeeflow approach
- expected business outcome

3. Customer challenge / background
- current business challenge
- operational pain points
- project drivers
- relevant constraints

4. Proposed Yeeflow solution
- what is being proposed
- how Yeeflow solves the problem
- why this is the right fit
- connection to workflows, data, integrations, dashboards, governance, or AI where relevant

5. Scope of work
- in-scope workstreams
- deliverables
- implementation boundaries
- responsibilities

6. Solution approach / implementation method
- discovery / design
- configuration / build
- testing / validation
- deployment / handover
- training / enablement if relevant

7. Timeline / phases
- project phases
- milestone timing
- dependencies if relevant

8. Assumptions
- customer readiness assumptions
- access assumptions
- technical assumptions
- governance / stakeholder assumptions

9. Exclusions / out of scope
- clearly state what is not included

10. Risks / dependencies
- major risks
- project dependencies
- customer-side dependencies

11. Effort / pricing summary
- high-level commercial summary if appropriate
- or reference to separate estimation/pricing sheet

12. Next steps
- approval path
- workshop / follow-up
- delivery readiness actions

Proposal messaging guidance
Frame Yeeflow around:
- practical business execution
- connected workflows and systems
- structured business applications
- governed AI where relevant
- repeatable delivery value
- scalable enterprise fit

Design guidance
Use the same Yeeflow visual system:
- clean spacing
- strong hierarchy
- subtle structure
- restrained color use
- premium but practical layout
- consistent Yeeflow logo and color rules

Review requirements
Before finalizing, check:
- Is the proposal easy for an executive or buyer to scan?
- Is the Yeeflow value proposition clear?
- Is the scope realistic and understandable?
- Are roadmap statements believable?
- Does the proposal look consistent with Yeeflow’s design system?
