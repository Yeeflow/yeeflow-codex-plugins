Yeeflow Event Brochure Standard

Purpose
Use this standard when creating or reviewing Yeeflow event brochures, product-introduction booklets, printable leave-behinds, and customer-facing handout materials.

This standard is intended for event and marketing materials that need to:
- introduce Yeeflow clearly
- support booth conversations
- summarize Yeeflow's platform value
- show practical current product proof
- drive demo booking or follow-up conversations

The brochure should feel like a compact printed extension of Yeeflow's event booth, product-introduction slide deck, and enterprise AI positioning.

Core positioning
Yeeflow should be presented as an enterprise AI application and execution platform.

Core message:
Build, use, and execute AI-powered business applications on one governed platform.

Core platform model:
Build. Use. Execute. Extend. Govern. Reuse.

1. Default brochure format

Default format:
A5 4-page booklet

Recommended production setup:
- Finished page size: A5 portrait, 148 x 210 mm
- Artwork size with bleed: 154 x 216 mm
- Bleed: 3 mm on all sides
- Resolution: 300 dpi
- Color direction: CMYK-ready
- Safe margin: minimum 8 mm; 10-12 mm preferred for logos, headlines, QR codes, and contact details
- Design review order: reader-order pages
- Final print imposition: confirm with printer

Recommended page sequence:
1. Page 1 - Front cover
2. Page 2 - Platform story
3. Page 3 - Product proof and use case
4. Page 4 - CTA and contact

For final print, confirm whether the printer wants:
- reader-order individual pages, or
- imposed spreads:
  - outside spread: Page 4 left + Page 1 right
  - inside spread: Page 2 left + Page 3 right

2. When to use A5 booklet instead of tri-fold

Use the A5 4-page booklet when:
- the material needs to explain Yeeflow's platform story
- visual diagrams need more breathing room
- the brochure should feel premium and executive-readable
- the content includes product proof and a practical use case
- the team wants a leave-behind that feels closer to a compact sales asset

Avoid A4 tri-fold when:
- the content becomes too dense in narrow panels
- diagrams become repetitive or difficult to read
- the story requires multiple strategic concepts
- the brochure needs a stronger premium event impression

3. Standard page structure

Page 1 - Front cover
Role:
Category statement and first impression.

Required content:
- Yeeflow brand/logo or clean logo-safe area
- Headline: Enterprise AI Application and Execution Platform
- Support line: Build, use, and execute AI-powered business applications on one governed platform.
- Event line, if relevant
- Company line, if relevant
- Optional label: Product introduction booklet

Visual direction:
- Dark premium background
- Controlled Yeeflow blue-led gradient
- Restrained purple/pink accents
- Strong whitespace
- One calm platform/execution visual anchor
- No dense feature detail

Page 2 - Platform story
Role:
Explain what Yeeflow is and how the platform model works.

Required content:
- Headline: Build. Use. Execute. Extend. Govern. Reuse.
- Support line: A practical platform model for turning business processes into AI-powered operations.
- Definition block: Yeeflow connects applications, workflows, data, AI, and integrations in one platform layer for real operational work.

Recommended platform model copy:
- Build application and process components faster.
- Use AI inside daily operational work.
- Execute configured workflows, services, and actions.
- Extend through models, APIs, OAuth, and custom connections.
- Govern with control, visibility, and enterprise discipline.
- Reuse assets across applications, teams, and environments.

Optional supporting labels:
- Applications
- Workflows
- Data
- AI
- Integrations

Visual direction:
- Light or balanced explanatory background
- One dominant platform diagram
- Six connected zones: Build, Use, Execute, Extend, Govern, Reuse
- Optional smaller supporting layer for Applications, Workflows, Data, AI, Integrations
- Clear hierarchy and print-readable labels
- Avoid repeated hub diagrams if another page already uses a similar structure

Page 3 - Product proof and use case
Role:
Ground the platform story in current product proof and a practical business scenario.

Required headline:
What's real now in Yeeflow AI

Required support line:
Current product progress supports practical AI-powered work across build, use, execution, connection, reuse, and extensibility.

Recommended five proof areas:
1. AI Builder
   Generate practical approval forms, data lists, fields, and workflow foundations for review.

2. AI inside apps
   Support analysis, summarization, classification, and guided next steps inside business applications.

3. Agent/Copilot actions
   Trigger configured actions, including HTTP request capability, within defined process boundaries.

4. API/OAuth connections
   Connect to models, services, HTTP APIs, OAuth-secured systems, and custom AI connections.

5. Reuse and extensibility
   Export, import, duplicate, move, and reuse AI assets; support advanced scenarios with custom code control and custom code action.

Recommended use case:
Supplier onboarding: from request to AI-assisted execution

Recommended journey strip:
Request -> AI Review -> Approval Workflow -> External Check -> Dashboard -> Reuse

Visual direction:
- Light explanatory background
- Calm proof-board style
- Five compact proof blocks or rows
- One simple use-case journey strip
- Short labels and readable text
- Avoid release-note styling
- Avoid dense product manuals or long feature explanations

Page 4 - CTA and contact
Role:
Drive demo booking and follow-up conversation.

Required headline:
Book a product demo

Recommended body copy:
Meet Yeeflow at Asia Tech x Singapore 2026 to explore practical AI execution for your organization.

Recommended CTA prompt:
Discuss your AI application and workflow use case.

QR label:
Scan to book a demo

Standard contact block:
- Contact Sales: sales@yeeflow.com
- Support Email: support@yeeflow.com
- Office Phone: +65 6359 2397
- Website: yeeflow.com

Visual direction:
- Dark premium or balanced closing page
- Visually echo the front cover
- Keep the page simple and action-oriented
- Top area: logo or logo-safe area
- Middle area: CTA headline and QR-safe area
- Lower area: short body copy and contact block
- No extra feature list

QR rules:
- Use a clean square QR placeholder or final QR code
- High contrast
- Minimum 5 mm quiet zone
- Do not place QR near trim, fold, or page edge
- Do not place QR over gradients, patterns, or complex visual effects
- Label clearly: Scan to book a demo

4. Visual style standard

Overall style:
- Premium
- Calm
- Structured
- Enterprise-ready
- Modern but restrained
- Product-led
- Credible
- Clean hierarchy
- Strong whitespace
- Print-readable
- Consistent with Yeeflow booth and event slide visuals

Color direction:
- Yeeflow blue as the anchor
- Restrained purple/pink accents
- Dark premium backgrounds for cover and CTA pages
- Light or balanced backgrounds for explanation and proof pages
- Avoid low-contrast text on gradients

Suggested palette:
- Primary blue: #146FF6
- Secondary purple: #7C4DFF
- Accent pink: #E85ECF
- Neutral dark: #333333
- White and soft neutral backgrounds

Typography direction:
- Use a clean modern sans-serif
- Large, readable page headlines
- Short body copy
- Strong hierarchy
- Avoid long paragraphs
- Avoid body text below approximately 7.5-8 pt
- Prefer 9-10 pt or larger for body text when possible

Diagram direction:
- Use simplified platform diagrams
- Use broad zones, large labels, and simple connectors
- Prefer vector-style graphics and clean icon-like shapes
- Avoid detailed screenshots unless specifically required and print-readable
- Avoid repeated diagrams that look too similar from page to page

5. Logo rules

Use official Yeeflow logo assets only.

Logo selection:
- Dark backgrounds: Yeeflow_logo_Whitex25.png
- Light backgrounds: Yeeflow_logo_Standardx25.png
- Symbol-only on dark backgrounds: Yeeflow_logo_only_white_2.png
- Symbol-only on light backgrounds: Yeeflow_logo_only_blue.png

Logo fidelity:
- Do not recreate, redraw, approximate, recolor, stretch, crop, rotate, outline, or restyle the Yeeflow logo.
- Do not generate fake Yeeflow logos.
- Do not place small fake logos inside diagrams or mock UI.
- If exact logo fidelity cannot be preserved in generated artwork, leave the logo area clean and empty.
- Add the official logo manually later when needed.
- Prefer one clear logo placement per page, not multiple small placements.

Logo-safe generation rule:
For AI-generated brochure artwork, if the model cannot use the exact logo file reliably:
- remove any generated logo
- keep the area clean
- do not add placeholder text
- do not add a dashed safe-area box in final artwork
- manually place the official logo during final production

6. Messaging rules

Use this positioning:
Enterprise AI Application and Execution Platform

Use this core message:
Build, use, and execute AI-powered business applications on one governed platform.

Use this platform model:
Build. Use. Execute. Extend. Govern. Reuse.

Use no-code carefully:
- No-code may be mentioned only as a supporting foundation if required.
- Do not lead with generic no-code positioning.
- Do not position Yeeflow as only a no-code platform with AI features.

Preferred language:
- practical AI execution
- real business systems
- governed platform
- configured workflows and actions
- approved connections
- operational visibility
- enterprise discipline
- reusable AI assets
- current product progress

Avoid:
- AI transformation without operational meaning
- fully autonomous enterprise app generation
- unrestricted autonomous execution
- complete multi-agent orchestration as current
- full governance, lifecycle, observability, memory, testing, certified packs, or marketplace claims as current unless explicitly approved
- internal roadmap terms in customer-facing artwork
- Q2/Q3/Q4 labels in the final brochure artwork
- roadmap direction or maturity direction wording in final customer-facing design

7. Product proof rules

Current proof can include:
- AI Builder for approval forms, data lists, fields, and workflow foundations
- AI inside apps for analysis, summarization, classification, and guidance
- Agent and Copilot actions, including HTTP request capability
- model options and custom AI connections
- HTTP API and OAuth connections
- export, import, duplicate, move, and reuse of AI assets
- custom code control and custom code action for advanced scenarios
- forms, workflows, data, dashboards, and Microsoft ecosystem foundations

Proof language should be:
- practical
- specific
- current
- credible
- concise
- customer-facing

Use careful wording:
- for review
- configured
- approved
- within defined process boundaries
- supports
- helps teams
- current product progress

Avoid:
- autonomous approval
- unrestricted system access
- invented performance metrics
- unsupported customer claims
- unsupported certifications or awards

8. Use case standard

Default event use case:
Supplier onboarding

Reason:
Supplier onboarding is executive-friendly, cross-functional, and relevant across industries. It naturally demonstrates forms, documents, AI review, workflow approvals, external checks, dashboards, and reuse.

Default use case headline:
Supplier onboarding: from request to AI-assisted execution

Default journey:
Request -> AI Review -> Approval Workflow -> External Check -> Dashboard -> Reuse

Use case rules:
- Keep the flow easy to understand in under 10 seconds.
- Show AI as assisting review, classification, summarization, or missing-information detection.
- Keep approvals and execution tied to configured workflows and controls.
- Show external checks as approved service or API calls.
- Do not imply autonomous supplier approval.

9. Print production standard

Page setup:
- A5 portrait
- Finished size: 148 x 210 mm
- Artwork with bleed: 154 x 216 mm
- 3 mm bleed on all sides
- 300 dpi raster effects/images
- CMYK-ready color setup
- Safe margin: minimum 8 mm, 10-12 mm preferred

Export:
- Export final artwork as print-ready PDF when approved
- Include bleed and crop marks if requested by printer
- Embed or outline fonts if sending to printer
- Preserve editable source file separately

Review before print:
- Check all text for spelling and final wording
- Confirm logo has been manually placed using official asset
- Confirm QR code is final and scannable
- Confirm contact details are correct
- Confirm no internal roadmap language is visible
- Confirm no fake customer logos, metrics, awards, or certifications
- Confirm all text remains readable at A5 size
- Confirm enough safe margin around all key content

10. Recommended workflow

Step 1 - Plan
Create the A5 booklet content plan first:
- page objectives
- headline and body copy
- visual direction
- proof points
- CTA/contact block

Step 2 - Approve copy
Review all text before generating artwork.

Step 3 - Generate artwork page by page
Recommended order:
1. Page 1 - Front cover
2. Page 4 - CTA/back cover
3. Page 2 - Platform story
4. Page 3 - Product proof and use case

Step 4 - Logo correction
If the generated artwork includes inaccurate logos:
- regenerate with logo removed, or
- manually remove the generated logo
- place official logo asset manually

Step 5 - QR and contact finalization
Add final QR code and confirm contact block:
- Contact Sales: sales@yeeflow.com
- Support Email: support@yeeflow.com
- Office Phone: +65 6359 2397
- Website: yeeflow.com

Step 6 - Final print production
Prepare:
- print-ready PDF
- editable source file
- individual page exports if needed
- imposed printer spread if requested
